## Project Overview
- meltica provides a provider-agnostic Go SDK that normalizes REST and WebSocket trading/data flows across major crypto exchanges. Core abstractions (`core/`) define the contract, provider adapters translate exchange-specific APIs to that contract, and tooling (`cmd/`, `internal/`) enforces schema and capability conformance.
- Typical workflow: implement adapters or features against `core` models, validate protocol assets with `conformance` + `protocol/`, enforce standards via `internal/meltilint`, exercise integrations with `internal/test` goldens, and ship binaries via `cmd/`.

## Module Roles
- `core`: canonical domain models, provider interfaces, capability bitset helpers, symbol/topic utilities, websocket event types.
- `errs`: unified error envelope (`errs.E`) and standardized capability errors.
- `transport`: shared HTTP client wrapper with signing hooks, retries, and token-bucket rate limiting.
- `protocol`: JSON Schemas, golden vectors, and protocol version constant that lock the wire format.
- `conformance`: harness that loads schemas, validates vectors, and asserts provider capability declarations.
- `internal/meltilint`: static analysis suite enforcing protocol standards, provider compliance, and documentation presence; includes CLI entrypoint.
- `internal/test`: golden/vector tests, fixture capture helpers, and provider-specific conformance checks using live or recorded data.
- `providers/<exchange>`: concrete adapters translating each exchange’s REST/WebSocket APIs, request signing, error/status mappings, and associated tests.
- `cmd`: CLI tools (`barista`, `market-stream`, `validate-schemas`) for scaffolding, streaming, and schema validation.
- `docs`: human-readable governance (expectations, how-to guides, migration steps, validation rules).
- `out`: compiled binaries generated by `make build` for distribution/testing.

## Workflow Summary
- Development centers on maintaining `core` interfaces and updating providers to honor capability contracts.
- Protocol changes must coincide with schema/vector updates in `protocol/`, validated via `conformance.ValidateSchemas` and guarded by `internal/meltilint`.
- Provider work typically starts with `cmd/barista` scaffolding, implementation in `providers/<exchange>`, tests under `internal/test`, then verification using `go test ./...` and running the lint + schema tools.
- Deployment/distribution leverages `Makefile` targets to compile binaries into `out/`, while `cmd/market-stream` serves as manual integration testing tool.

## 1. Project‑wide Purpose

Meltica is a **Go‑based SDK** that provides:

1. **Unified access** to multiple crypto‑exchange APIs (Binance, Coinbase, Kraken, OKX, etc.).  
2. **Real‑time market‑data streaming** (WebSocket) and RESTful data retrieval.  
3. **Schema‑driven validation** of provider responses against a common protocol.  
4. **Linting & conformance tooling** to ensure provider implementations meet the protocol’s expectations.  
5. **Command‑line utilities** for generating schemas, validating them, and running example clients.

The library is split into logical modules (cmd, core, providers, protocol, transport, etc.) that each encapsulate a distinct responsibility.

## 4. Workflow & Build Process

### 4.1. Building

| Target | Description |
|--------|-------------|
| `make build` | Builds all binaries (`cmd/*`, `market-stream`, `meltilint`). |
| `make out` | Generates the `out/` directory with binaries, docs, and lint reports. |
| `make release` | Builds platform‑specific binaries and tags a release. |

### 4.2. Running the Library

```bash
# Example: Run the market‑stream CLI for Binance
./cmd/market-stream/market-stream --provider binance --topic ticker
```

The CLI uses the `cmd/market-stream/main.go` entry point, which internally loads the requested provider, subscribes to the specified topic, and streams events to stdout.

### 4.3. Testing

| Test Type | Command | What it verifies |
|-----------|---------|------------------|
| **Unit tests** | `go test ./...` | Core logic, numeric helpers, provider REST/WS handlers. |
| **Golden‑file tests** | `go test ./internal/test` | Ensures provider responses match stored JSON vectors. |
| **Conformance tests** | `go test ./conformance` | Runs the harness against all providers to verify protocol compliance. |
| **Linting** | `make lint` | Runs `meltilint` against provider packages. |
| **Schema validation** | `make validate-schemas` | Validates JSON schemas against the protocol. |

### 4.4. Linting & Conformance

- **meltilint** (`internal/meltilint`) is a custom linter that checks provider packages for:
  - Correct struct tags (`json`, `validate`).
  - Proper error handling.
  - Documentation presence.
  - Numeric precision (decimals vs floats).
  - Protocol‑specific field names.
- The linter is invoked via `make lint` or directly with `./cmd/meltilint/meltilint`.

### 4.5. Documentation

- `docs/` contains Markdown guides for expectations, migration, and provider implementation.
- `protocol/README.md` explains the JSON schema protocol.
- `Makefile` has a `docs` target that generates Markdown from the `protocol` schemas.

### 4.6. Command Line Tools

The `cmd/` directory contains command-line tools that demonstrate SDK usage:

- **market-stream**: Real-time market data streaming from exchanges
- **barista**: Generate new provider scaffolds
- **validate-schemas**: Validate protocol schemas and golden vectors

These tools serve as practical examples and are useful for quick testing and integration validation.

## 5. High‑Level Workflow

1. **Provider Implementation**  
   - Add a new provider under `providers/<name>/`.  
   - Implement REST and WebSocket endpoints, signing, status handling, and error types.  
   - Follow the guide in `docs/how-to/implementing-a-provider.md`.

2. **Schema Generation**  
   - Run `make validate-schemas` to ensure provider schemas match the protocol.  
   - Use `cmd/validate-schemas/main.go` to validate JSON schemas against the protocol definitions.

3. **Linting & Conformance**  
   - Run `make lint` (or `./cmd/meltilint/meltilint`) to lint provider code.  
   - Run `make test` to execute unit, golden‑file, and conformance tests.

4. **Building Binaries**  
   - `make build` compiles the CLI tools (`market-stream`, `barista`, `meltilint`).  
   - Binaries are placed in `out/` and optionally pre‑built binaries are shipped in `market-stream/`.

5. **Release**  
   - Tag a new release with `make release`.  
   - The Makefile will bump the protocol version, rebuild binaries, and push tags.

---

## 6. Summary

- **Core SDK** (`core/`) provides the building blocks for all providers.  
- **Provider packages** (`providers/*/`) implement exchange‑specific logic.  
- **Protocol** (`protocol/`) defines the JSON schema contract and sample vectors.  
- **Transport** (`transport/`) abstracts HTTP/WebSocket communication and rate‑limiting.  
- **Linting & conformance** (`internal/meltilint/`) ensures provider code adheres to the protocol.  
- **CLI tools** (`cmd/`) expose the SDK's functionality to end‑users and demonstrate typical usage patterns.  
- **Testing** is heavily driven by golden‑file vectors and conformance harnesses.

The project is designed to be **extensible**: adding a new provider is a matter of creating a new package under `providers/`, implementing the required interfaces, and running the linter and conformance tests. The CLI tools allow developers to quickly spin up market‑data streams or validate schemas without writing any code.
